# Lesson 02: Logging Fundamentals

## Learning Objectives

By the end of this lesson you should be able to:

- Explain logging.
- Design useful logs.
- Understand structured logging.
- Understand log levels.
- Design centralized logging systems.
- Avoid common logging mistakes.

---

## Introduction

Logs are usually the first place engineers look during incidents.

Example:

User reports:

    Upload failed.

Question:

Why?

Logs often contain the answer.

Without logs:

Troubleshooting becomes guesswork.

---

## What Is A Log?

A log is a recorded event generated by a system.

Examples:

    User logged in

    Payment succeeded

    File uploaded

    Workflow failed

Logs create a history of system behavior.

---

## Why Logs Matter

Logs help:

- debugging
- auditing
- troubleshooting
- incident response
- security investigations

Production systems generate large numbers of logs.

---

## Good Log Example

    User 123 uploaded file abc.jpg

Useful.

Contains context.

---

## Bad Log Example

    Error occurred

Provides little value.

Avoid vague logs.

---

## Log Levels

### DEBUG

Detailed diagnostic information.

Example:

    Validating JWT token

Usually disabled in production.

---

### INFO

Normal operations.

Example:

    User created

    Workflow started

---

### WARN

Unexpected but recoverable.

Example:

    Retry attempt 2

---

### ERROR

Operation failed.

Example:

    Payment failed

---

### FATAL

System cannot continue.

Example:

    Database unavailable

---

## Structured Logging

Bad:

    User 123 created workspace 456

Good:

    {
      "event":"workspace_created",
      "user_id":123,
      "workspace_id":456
    }

Benefits:

- searchable
- machine readable
- easier analysis

---

## Correlation IDs

Distributed systems require request tracking.

Example:

Request ID:

    req_abc123

Every log includes:

    req_abc123

Now logs can be connected together.

---

## Example Flow

Client

↓

API

↓

Redis

↓

PostgreSQL

All components log:

    req_abc123

This allows end-to-end troubleshooting.

---

## Centralized Logging

Bad:

Logs stored on individual servers.

Good:

All logs collected centrally.

Architecture:

Applications

↓

Log Collector

↓

Log Storage

↓

Dashboards

## Security Considerations

Never log:

- passwords
- tokens
- secrets
- private keys

Logs often live for years.

Protect sensitive data.

## Failure Scenarios

---

### Scenario 2

---

### Scenario 3

Sensitive secrets logged.
